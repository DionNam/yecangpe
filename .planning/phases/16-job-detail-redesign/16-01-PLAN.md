1. Create SQL migration `supabase/migrations/20260207_backfill_slugs.sql`:
   - Create a PL/pgSQL function `generate_job_slug(p_title TEXT, p_id UUID)` that:
     - Converts title to lowercase, replaces non-alphanumeric with hyphens, trims leading/trailing hyphens, collapses multiple hyphens
     - Extracts 8-char suffix from UUID (remove dashes, take first 8)
     - Returns `{slug}-{suffix}` or just `{suffix}` if title produces empty slug (Korean-only titles)
   - Run `UPDATE job_posts SET slug = generate_job_slug(title, id) WHERE slug IS NULL;`
   - Drop the function after use (it's only for backfill; JS generateJobSlug handles future posts)

2. In `apps/web/app/actions/jobs.ts` `createJobPost`:
   - Import `generateJobSlug` from `@repo/lib`
   - After the successful insert, we need the post ID. Change the insert to use `.select('id')` to get the new post's ID
   - Then update the slug: `await supabase.from('job_posts').update({ slug: generateJobSlug(result.data.title, newPost.id) }).eq('id', newPost.id)`
   - Alternatively, add slug to the insert directly by generating a temporary UUID first. Simpler approach: use `.insert({...}).select('id').single()` then update slug.

3. In `apps/admin/app/actions/posts.ts`:
   - Find the admin createPost function that inserts into job_posts
   - Import `generateJobSlug` from `@repo/lib`
   - Apply same pattern: insert, get ID, update slug
  </action>

1. Create `apps/web/app/(main)/jobs/[slug]/page.tsx`:
   - Server component (no 'use client')
   - Accept `params: Promise<{ slug: string }>`
   - Fetch job by slug: `.from('job_posts').select('*').eq('slug', slug).eq('review_status', 'published').single()`
   - If not found, check if slug is a UUID format (regex test). If so, look up by ID and redirect to `/jobs/${job.slug}`. If still not found, call `notFound()`.
   - **DO NOT require authentication to view the page** (for SEO crawlability). Auth check remains only for gating like/apply actions (handled in subsequent plans).
   - Increment view count: `await supabase.rpc('increment_view_count', { post_id: job.id })`
   - Fetch metrics config, like count, check if user is logged in (optional auth), check if user liked post
   - Calculate display metrics with `getDisplayMetrics`
   - Render the existing `<JobDetail>` component with same props (this will be replaced in Plan 02, but keep it working for now)
   - Keep the back link, wrapper div, etc.

2. Convert `apps/web/app/(main)/jobs/[id]/page.tsx` to a pure redirect:
   - Server component that looks up job by ID, gets its slug, and redirects to `/jobs/${slug}`
   - If not found, call `notFound()`
   - DO NOT increment view count here (avoid double-increment per Pitfall 7 in research)

3. Update all internal links to use `job.slug` instead of `job.id`:
   - `apps/web/components/jobs/job-card.tsx`: Change `router.push(\`/jobs/${job.id}\`)` to `router.push(\`/jobs/${job.slug || job.id}\`)`

   - `apps/web/components/employer/my-posts-table.tsx`: Change `href={\`/jobs/${post.id}\`}` to use slug
   - Use fallback pattern `job.slug || job.id` to handle any posts that might still have NULL slug

NOTE: The job list page query (`apps/web/app/(main)/jobs/page.tsx`) already selects `*` which includes slug. Verify the select includes slug field -- if it selects specific columns, add `slug` to the select list.
